# 17 — 개발 지원 (더미 시드 · 진단 · 클린업)

> 작성일: 2026-04-24
> 상태: v1 구현 완료 (로컬 한정)

---

## 1. 왜 만들었나 (Problem)

- 데모/개발 환경에서 기능 시연을 위해 **의미 있는 데이터** 가 항상 필요하다.
  예: 지난 달 수령된 책이 있어야 독후감 기능을 시연할 수 있음.
- 반복적으로 DB 를 수작업으로 채우는 것은 비효율적이고 리그레션 대응이 어렵다.
- 운영 환경에서는 이 데이터들이 **절대로 생성되면 안 된다**.

## 2. 목표 (Goal)

- 부팅 시 자동으로 클럽/회원/책 신청/주문/독후감 시드를 넣어 기능이 "살아있는" 상태로 뜨게 한다.
- 스키마 수정/데이터 오염 시 빠르게 초기화할 일회성 러너 제공.
- 운영 프로파일에서는 모두 **비활성**.

## 3. 정책 / 규칙 (Rules)

- **가드**: 모든 시드/러너는 프로파일 또는 환경 플래그로 보호.
  - `DummyDataSeeder` — 로컬/개발 프로파일에서만 실행. 프로덕션 부팅 시 no-op.
  - `OrdersCleanupRunner` — 명시 플래그로만 수행 (삭제성 작업).
- **멱등성**: 이미 있는 데이터는 덮어쓰지 않음. 없는 것만 추가.
- **포함 범위** (DummyDataSeeder):
  - 기본 클럽(무제) + DEVELOPER 1 + ADMIN 1 + MEMBER N.
  - 최근 2~3 개월치 BookRequest (각 상태 분포) + Order + BookReport 일부.
  - `AppConfigSeeder` 와 협력해 기본 키들(화이트리스트, 기본 예산) 시드.
- **진단 엔드포인트**: (별도 컨트롤러가 있다면) 상태 점검/헬스 체크. 프로덕션 노출 시 `management.endpoints.web.exposure` 레벨로 제어.

## 4. UX 흐름

개발자용이므로 사용자 UX 없음. 로컬에서 `./gradlew bootRun` 하면:

1. DB 가 비어 있으면 → 시드 실행 → 로그에 "Dummy data seeded (N clubs / M members / …)".
2. 이미 데이터 있으면 → "Skip seeding (already populated)" 로그.
3. 이상 상태일 땐 `OrdersCleanupRunner` 를 플래그로 돌려 `orders/order_items` 정리.

## 5. API 표면

- 진단(있을 경우) 은 `/api/v1/developer/**` 아래에 배치. 로컬 한정.
- 시드는 API 아님 (ApplicationRunner).

## 6. 수용 기준 (Acceptance)

- [x] 로컬 부팅 시 시연 가능 상태 자동 구성.
- [x] 운영 프로파일에서 시드가 실행되지 않음.
- [x] 시드 멱등 — 재부팅해도 중복 생성 안 됨.
- [ ] 시드 데이터 구성이 docs/plan/과 일치(월별 예산/나눔/미제출 등 데모 장면 모두 커버) — 점검 필요.
- [ ] 진단 엔드포인트의 접근 정책 문서화.

## 7. 오픈 이슈 / 향후 계획

- **시나리오별 프리셋**: "월 마감 직후", "나눔 수락 대기 다수" 같은 데모용 프리셋 선택.
- **테스트 격리**: 통합 테스트에서 사용할 별도 시드 세트.
- **E2E 시드**: Playwright 에서 호출하는 전용 reset 엔드포인트 (로컬 한정).
- **운영 관리자용 진단**: 부팅/세션/큐 상태 보기 — DEVELOPER 만 접근.
